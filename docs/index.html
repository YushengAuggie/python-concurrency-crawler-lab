01 Thread pool, Future, and waiting for progress
Learn

A thread pool runs blocking fetch functions in worker threads. submit() returns a Future, which is a handle to work that may still be running.

The crawler stores pending futures so it knows what is in flight, waits for the first one to finish, calls result() to observe failures, then submits more URLs.

Run
python3 compare_examples.py

# Read:
ThreadPoolWebCrawler.run()
wait(..., FIRST_COMPLETED)

Open the Future/wait loop

Check yourself: Why does the crawler submit only up to the available worker slots instead of submitting every discovered URL immediately?

02 Asyncio Task, await, and the event loop
Learn

Asyncio schedules Task objects on one event loop thread. A task pauses at await, leaving the loop free to advance other tasks while a simulated request waits.

The async crawler owns shared state in one coordinator method, so completed tasks return links and the coordinator updates the queue without a thread lock.

Run
python3 async_crawler.py
python3 compare_examples.py

# Read:
AsyncWebCrawler.run()
asyncio.wait(..., FIRST_COMPLETED)

Open the asyncio run loop

Check yourself: Why can completion order differ from sequential BFS order, even if every reachable URL is still found?

Shared boundary: the same offline site exposes a blocking fetch and an async fetch, so the comparison is fair.
def get_html_content(url: str) -> str:
    time.sleep(PAGE_DOWNLOAD_DELAYS.get(url, 0.02))
    return _read_example_page(url)

async def get_html_content_async(url: str) -> str:
    await asyncio.sleep(PAGE_DOWNLOAD_DELAYS.get(url, 0.02))
    return _read_example_page(url)
Threads: submit() returns a Future; wait() pauses coordination until at least one fetch is done.
future = thread_pool.submit(self.crawl_page, url)
self.pending_futures.add(future)

done_futures, pending_futures = wait(
    self.pending_futures,
    return_when=FIRST_COMPLETED,
)
for future in done_futures:
    future.result()
Asyncio: tasks advance on the event loop and yield while the simulated request waits.
task = asyncio.create_task(self.fetch_links(url))
pending_tasks.add(task)

done_tasks, pending_tasks = await asyncio.wait(
    pending_tasks,
    return_when=asyncio.FIRST_COMPLETED,
)
for task in done_tasks:
    url, discovered_links = task.result()

Self-check

Can you predict the behavior?

Answer before opening the code again. Correct explanations matter more than memorizing syntax.

Why not call requests.get() directly inside async def?

Explain in your own words

Asyncio needs each task to yield at await. A blocking call keeps the event loop thread busy, so unrelated tasks cannot make progress.

What does max_concurrency=2 limit?

Explain in your own words

It limits active fetches, not total discovered pages. A crawler may still discover many URLs and hold them in the waiting queue.

What does pool.submit(crawl_page, url) return immediately?

Explain in your own words

A Future lets the coordinator remember scheduled work, wait for completion, and call result() to surface unexpected worker exceptions.

After cancelling pending tasks, why await them?

Explain in your own words

Cancellation is delivered into the coroutine. Awaiting gives cancelled tasks a chance to run cleanup and lets the caller observe their final state.
