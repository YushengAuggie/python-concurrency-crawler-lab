02 Asyncio Task、await 与 event loop
学习

Asyncio 会把 Task 调度到一个 event loop thread 上。Task 在 await 处暂停，让 event loop 有机会推进其他正在等待的任务。

这个 async crawler 只在一个 coordinator 方法里更新共享状态；完成的 task 返回链接，由 coordinator 统一入队，所以不需要线程锁。

Threads：submit() 返回 Future；wait() 让协调逻辑等待至少一次抓取完成。
future = thread_pool.submit(self.crawl_page, url)
self.pending_futures.add(future)

done_futures, pending_futures = wait(
    self.pending_futures,
    return_when=FIRST_COMPLETED,
)
for future in done_futures:
    future.result()
Asyncio：tasks 在 event loop 中推进，并在模拟请求等待期间让出控制权。
task = asyncio.create_task(self.fetch_links(url))
pending_tasks.add(task)

done_tasks, pending_tasks = await asyncio.wait(
    pending_tasks,
    return_when=asyncio.FIRST_COMPLETED,
)
for task in done_tasks:
    url, discovered_links = task.result()

自查

你能预测行为吗？

先回答，再回去查看代码。能够解释原因，比记住语法更重要。

为什么不能在 async def 中直接调用 requests.get()？

用自己的话解释

Asyncio 需要 task 在 await 处让出控制权。阻塞调用会占住 event loop 线程，导致其他 tasks 无法继续推进。

max_concurrency=2 限制的是什么？

用自己的话解释

它限制的是活跃 fetch 数量，不是已发现页面总数。Crawler 仍可能发现很多 URL，并把它们保存在等待队列中。

pool.submit(crawl_page, url) 会立即返回什么？

用自己的话解释

Future 让 coordinator 记住已提交工作，等待它完成，并通过 result() 暴露 worker 中的意外异常。

取消 pending tasks 后，为什么还需要 await 它们？

用自己的话解释

Cancellation 会被送进 coroutine 内部。继续 await 可以让 task 执行清理逻辑，也让调用方观察它们的最终状态。
